## Why SD-WAN Security Is Different

Traditional WAN security assumes traffic flows through a central hub (headquarters or data center) where inspection happens. SD-WAN shatters that model — traffic can break out locally, cloud-to-cloud, or peer-to-peer. This creates inspection gaps that attackers exploit.

The Fortinet SD-WAN security model addresses this through **Security-Driven Networking**: security controls are embedded at every edge point, enforced by the Security Fabric, and centrally visible through FortiManager and FortiAnalyzer.

```
[Branch] ─── IPSec Overlay ──→ [Hub / FortiGate] ─── [Internet]
```

## Zero-Trust Segmentation with SD-WAN

```bash
# FortiGate CLI: Configure ZTNA access proxy for internal application
config firewall access-proxy
    edit "corporate-app-proxy"
        set vip "corporate-app-vip"
        set client-cert enable
        config api-gateway
            edit 1
                set url-map "/app"
                set service "tcp"
                set realservers
                    edit 1
                        set ip 10.0.1.100
                        set port 8443
                        set health-check "http-check"
                    next
                end
                set application-check enable
            next
        end
    next
end

# Create ZTNA policy requiring device compliance + user identity
config firewall policy
    edit 0
        set name "ZTNA-Corporate-App"
        set srcintf "ssl.root"
        set dstintf "internal"
        set srcaddr "ZTNA-verified-users"
        set dstaddr "corporate-app-vip"
        set service "HTTPS"
        set action accept
        set ztna-status enable
        set ztna-ems-tag "Compliant-Device"
        set groups "Domain-Users"
        set logtraffic all
        set utm-status enable
        set profile-group "strict-utm-profile"
    next
end
```

## Application Steering Security

A common SD-WAN misconfiguration: application steering rules that bypass security inspection for performance. Never do this:

```bash
# ❌ INSECURE: Steering direct to internet without inspection
config router policy
    edit 0
        set input-device "port1"
        set src "10.0.0.0/8"
        set dst "0.0.0.0/0"
        set action permit
        set output-device "wan1"
        # Missing: no security profile applied
    next
end

# ✅ SECURE: Always apply security profiles to steered traffic
config firewall policy
    edit 0
        set name "SD-WAN-Internet-Breakout"
        set srcintf "internal"
        set dstintf "sd-wan"
        set srcaddr "Branch-Users"
        set dstaddr "all"
        set action accept
        set utm-status enable
        set profile-group "internet-breakout-profile"  # Must include AV, IPS, AppControl, WebFilter
        set ssl-ssh-profile "deep-inspection"
        set logtraffic all
        set nat enable
    next
end
```

## SSL/TLS Deep Inspection

SSL inspection is essential but requires careful configuration to avoid breaking legitimate traffic:

```bash
# Create a certificate inspection profile for high-sensitivity categories
config firewall ssl-ssh-profile
    edit "banking-exemptions"
        set comment "Full inspection with banking/healthcare exemptions"
        config https
            set ports 443
            set status deep-inspection
            set unsupported-ssl-version block
            set min-allowed-ssl-version tls-1.2
        end
        config ssl-exempt
            edit 1
                set type category
                set category banking
            next
            edit 2
                set type category
                set category health-and-medicine
            next
        end
        config ssl-server
            edit 1
                set ip 10.0.1.0/24  # Internal servers
                set ssl-other-versions allow
            next
        end
    next
end
```

## Security Fabric Integration

The Security Fabric enables coordinated threat response across all security components:

```bash
# Enable Security Fabric on root FortiGate
config system csf
    set status enable
    set group-name "Acme-Security-Fabric"
    set group-password [encrypted]
    set management-ip 10.0.0.1
    set management-port 8013
    set saml-configuration-sync enable
    set downstream-access enable
    config trusted-list
        edit 1
            set authorization-type serial
            set serial "FGT60FTXXXXXXXX"
            set name "Branch-FortiGate-01"
        next
    end
end

# Automated threat response
config system automation-trigger
    edit "Critical-IPS-Alert"
        set event-type ips-logs
        set ioc-level critical
    next
end

config system automation-action
    edit "Block-Attacking-IP"
        set action-type quarantine
        set quarantine target-segment
        set ip-pool "quarantine-pool"
        set duration 3600
    next
end

config system automation-stitch
    edit "Auto-Quarantine-Critical-IPS"
        set trigger "Critical-IPS-Alert"
        set actions "Block-Attacking-IP"
    next
end
```

## Common NSE 7 Exam Topics (From Experience)

Having passed NSE 7 SD-WAN, these are the highest-weight topics:

1. **Performance SLA and Dead Interval** — understand how FortiGate declares a link dead and the math behind SLA thresholds
2. **SD-WAN Rules Priority** — service-based rules take precedence over application-based; understand the evaluation order
3. **Overlay and Underlay Architecture** — be able to explain the full packet flow from branch to hub, including ESP/IKE phase 1/2
4. **BGP over IPSec** — IBGP vs EBGP for hub-spoke; route redistribution into OSPF
5. **ZTNA vs VPN** — functional differences, deployment scenarios, and when to use each
6. **Security Profiles on SD-WAN Zones** — NGFW mode vs Profile-based; how profiles apply across zones

## Summary

Fortinet SD-WAN security requires treating every branch as a potential attack surface — applying full UTM inspection to all traffic (including SaaS breakouts), implementing Zero-Trust Network Access for application access, leveraging Security Fabric for automated threat response, and centralizing visibility in FortiAnalyzer. The most common mistake is trading security for performance on SD-WAN breakout policies — always apply security profiles, even if it costs a few milliseconds.
